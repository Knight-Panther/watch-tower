Clear recommendation: build Stage 2 semantic dedup first, starting with pgvector schema + embeddings pipeline + similarity filtering, then add basic human approval UI hooks for stage 4 later.

**Ranked priorities (value, cost efficiency, dependencies)**  
- **1) Stage 2: Embeddings + pgvector + semantic dedup**: High value (cuts noise early), cost-efficient (cheap filter before LLM), unblocks Stage 3/4 by providing similarity_score and embedding data.  
- **2) Stage 3: LLM brain (summary + importance score)**: High value but higher cost; should consume post-dedup articles only.  
- **3) Stage 4: Distribution pipeline + approval flow**: Value depends on clean upstream; best after Stage 2/3 are stable.  
- **4) Frontend approval UI**: Useful but depends on distribution events + LLM metadata.

**Top recommendation details: Stage 2 semantic dedup**

Core files to create  
- `packages/embeddings/src/index.ts` (public API)  
- `packages/embeddings/src/provider/*` (provider-specific embedder)  
- `packages/embeddings/src/queue.ts` (BullMQ producer/consumer)  
- `packages/embeddings/src/similarity.ts` (pgvector query helpers)  
- `packages/embeddings/src/schema.ts` (db migrations/DDL or adapter)  
- Worker: `apps/workers/src/embeddings.worker.ts` (or analogous in your structure)  
- Queue wiring: `apps/workers/src/queues.ts` update to register embeddings worker  
- Migration: `db/migrations/*` add `vector` column + `pgvector` extension

Key implementation decisions  
- **Embedding model**: pick a stable, low-cost option (e.g., OpenAI `text-embedding-3-small` or local option later).  
- **Vector size**: match model dimension; lock it in schema.  
- **Similarity metric**: cosine distance via pgvector; define threshold (e.g., 0.85–0.92) and tune.  
- **Dedup policy**:  
  - Compare new article against recent window (e.g., last 7–30 days) to reduce cost.  
  - Set `similarity_score` and mark duplicates (or link to canonical article).  
- **Queue design**: run embeddings worker immediately after ingest; enqueue by article id.  
- **Schema**:  
  - Add `embedding vector(N)`  
  - `similarity_score float` already exists  
  - Add an index: `ivfflat` or `hnsw` for vector search later (start with `ivfflat` if you want cheaper setup).

Estimated complexity  
- **M**: new package + schema + worker + basic query logic.

Gotchas / risks  
- **pgvector extension**: must be installed and enabled; deployment environment needs it too.  
- **Vector dimension mismatch**: must match model.  
- **Similarity threshold tuning**: too low → duplicates slip through; too high → false merges.  
- **Cost control**: embeddings per article; make sure to avoid re-embedding duplicates or unchanged content.  
- **Backfill**: old articles without embeddings may need a batch job.

**Alternative approaches worth considering**  
- **Cheap heuristic pre-dedup**: run a simple TF-IDF or title+domain hash before embeddings to cut costs further.  
- **LLM-assisted dedup**: use a small LLM for borderline similarity only; likely overkill initially.  
- **Local embeddings**: reduce cost and dependency (e.g., `bge-small`), but more infra.  
- **Skip vector index initially**: for small dataset, do linear scan with a time window; add index later.
