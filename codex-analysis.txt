Here’s the semantic dedup flow and edge‑case analysis with line references.

**Flow**
- Ingestion → queue: articles start at `pipeline_stage = 'ingested'` (implied by claim query) and are claimed in batches via `UPDATE ... FOR UPDATE SKIP LOCKED` to `pipeline_stage = 'embedding'` in `packages/worker/src/processors/semantic-dedup.ts:45-59`.
- Short content bypass: items with <10 chars are skipped and immediately moved to `pipeline_stage = 'embedded'` in `packages/worker/src/processors/semantic-dedup.ts:70-93`.
- Embedding generation: `embedBatch` for the batch and failure resets back to `ingested` in `packages/worker/src/processors/semantic-dedup.ts:99-115`.
- Dedup check: for each article, `findSimilarArticles` searches older non‑duplicate content; if found, the article is marked `duplicate` and linked, otherwise it is marked `embedded` in `packages/worker/src/processors/semantic-dedup.ts:117-165` using SQL from `packages/embeddings/src/similarity.ts:56-74`.
- Non‑duplicates are queued for LLM scoring in `packages/worker/src/processors/semantic-dedup.ts:172-179`.

**Race Conditions / Edge Cases**
- Same batch, identical articles: the loop is sequential (`await` per item), so later items can see earlier items once they are updated to `embedded` (`packages/worker/src/processors/semantic-dedup.ts:121-165`). This works only when `created_at` is strictly increasing. If two articles share the same `created_at`, the filter `created_at < current` in `packages/embeddings/src/similarity.ts:52-54` prevents a match, so both can be marked unique (false negative).
- Same batch, older/newer ordering inversion: the dedup query only matches “older” by `created_at` (`packages/embeddings/src/similarity.ts:52-54`). If the newer article gets an earlier `created_at` (clock skew, source timestamp), it won’t match even if the older was embedded.
- Concurrent workers: both workers claim articles into `pipeline_stage = 'embedding'` (`packages/worker/src/processors/semantic-dedup.ts:45-59`). The similarity query excludes `pipeline_stage = 'embedding'` (`packages/embeddings/src/similarity.ts:65-67`), so if Worker A checks before Worker B finishes its update, neither sees the other and both can be marked `embedded` (false negatives). This is a classic race between “check” and “update” across workers.
- `is_semantic_duplicate = false` filter: if this column can be `NULL`, those rows are excluded because `NULL = false` is unknown, which can hide legitimate candidates (`packages/embeddings/src/similarity.ts:65-68`).
- 30‑day cutoff: `created_at > cutoffDate` (`packages/embeddings/src/similarity.ts:68`) ignores older articles, so duplicates against older content are missed.

**Specific Scenarios**
- Two identical articles in the SAME batch:
  - If `created_at` differs (older/newer), the newer one should match the older after the older is embedded (`packages/worker/src/processors/semantic-dedup.ts:121-165`).
  - If `created_at` is equal, the strict `<` filter blocks the match (`packages/embeddings/src/similarity.ts:52-54`), so both can pass as unique.
- Two identical articles processed by DIFFERENT worker instances concurrently:
  - If both are in `pipeline_stage = 'embedding'` when the other checks, the query excludes them (`packages/embeddings/src/similarity.ts:65-67`), so both can pass as unique. If one finishes update before the other query and has an older `created_at`, it may be caught, but this is timing‑dependent.

**Gaps Leading to False Negatives**
- Strict `created_at < current`: misses ties and out‑of‑order timestamps (`packages/embeddings/src/similarity.ts:52-54`).
- Exclusion of `embedding` stage: concurrent workers won’t see each other during processing (`packages/embeddings/src/similarity.ts:65-67`).
- `maxAgeDays` window: duplicates older than 30 days are ignored (`packages/embeddings/src/similarity.ts:68`).
- Potential `NULL` for `is_semantic_duplicate`: `NULL` rows are excluded (`packages/embeddings/src/similarity.ts:65-68`).

**SQL WHERE Clause Review**
- `pipeline_stage NOT IN ('duplicate','rejected','ingested','embedding')` in `packages/embeddings/src/similarity.ts:65-67` is the primary source of concurrent‑worker misses. If you want cross‑worker detection, you need a way to include in‑flight items once their embeddings are available (or avoid the race entirely).
- `created_at < currentArticleCreatedAt` in `packages/embeddings/src/similarity.ts:52-54` causes tie failures. It should likely be `<=` with a tie‑breaker (e.g., `created_at < ... OR (created_at = ... AND id < ...)`) to ensure deterministic “older” selection.
- `is_semantic_duplicate = false` in `packages/embeddings/src/similarity.ts:65-68` can hide rows if the column is nullable; consider `IS NOT TRUE` or a default `false` with `NOT NULL`.

**Recommendations**
- Add a deterministic tie‑breaker: change the “older” filter to handle equal timestamps (e.g., `created_at < $ts OR (created_at = $ts AND id < $id)`) so same‑timestamp duplicates are caught; update `packages/embeddings/src/similarity.ts:52-54`.
- Mitigate concurrent‑worker races: options include
  1) two‑phase update: write embeddings first (still `embedding`), then run dedup query that includes `embedding` rows where `embedding IS NOT NULL`  
  2) move dedup into a single SQL transaction with explicit advisory locking by article hash  
  3) add a post‑embedding reconciliation pass that checks newly embedded rows against recent embeddings.
- Relax `is_semantic_duplicate` filter to handle NULLs if possible (`IS NOT TRUE`) or enforce `NOT NULL DEFAULT false`.
- Consider expanding or making `maxAgeDays` configurable depending on desired dedup scope.
