Timezone flow report (scheduled posting)

1) ScheduleModal creates Date from local inputs
- `packages/frontend/src/components/ScheduleModal.tsx`: `const scheduledAt = new Date(`${date}T${time}`);`
  - This parses a local‑time date/time string (no timezone), so the JS Date represents the intended local wall‑clock time.
- **Bug risk:** the initial `date` value is derived from `defaultDate.toISOString().split("T")[0]`, which uses UTC. If the user is in a negative/positive offset and local date differs from UTC, the date picker defaults to the wrong day (off‑by‑one). This only affects the initial default, but it can cause users to schedule the wrong date unless they notice. File: `packages/frontend/src/components/ScheduleModal.tsx`

2) How Date is converted when sent to API
- `packages/frontend/src/pages/Articles.tsx`: `scheduled_at: data.scheduledAt.toISOString()`
  - This converts the local Date to a UTC ISO string with `Z`. This is correct for unambiguous transmission, but it means the API always receives UTC timestamps.

3) How API stores it in PostgreSQL TIMESTAMPTZ
- `packages/api/src/routes/articles.ts`: `const scheduledAt = scheduled_at ? new Date(scheduled_at) : new Date();`
  - `scheduled_at` is expected to be ISO with timezone. With the frontend’s `toISOString()`, JS parses it as UTC correctly.
  - Drizzle inserts `scheduledAt` into `post_deliveries.scheduled_at` (TIMESTAMPTZ). Postgres stores it as an absolute timestamp; timezone handling is consistent.
- **Potential edge case (not currently triggered by frontend):** if any client sends `scheduled_at` without a timezone (e.g., `2026-02-01T10:00`), `new Date()` will interpret it in the API server’s local timezone, which could shift the instant.

4) How worker compares scheduled_at <= NOW()
- `packages/worker/src/processors/maintenance.ts`: SQL uses `WHERE scheduled_at <= NOW()` with `scheduled_at` TIMESTAMPTZ.
  - `NOW()` returns a TIMESTAMPTZ in the DB session timezone; comparison is on absolute instants, so it matches what was stored. No timezone mismatch here.

Summary of findings
- **Bug:** Default date picker value uses UTC date (`toISOString()`), which can be off by one day relative to the user’s local date. This can cause accidental mis‑scheduling if the user accepts defaults. File: `packages/frontend/src/components/ScheduleModal.tsx`
- **No issue:** Local date/time → `toISOString()` → `new Date(iso)` → TIMESTAMPTZ → `scheduled_at <= NOW()` is consistent and compares correct instants.
- **Potential external‑client risk:** If any non‑frontend client submits `scheduled_at` without timezone, API may misinterpret it in server local timezone. This is not happening with the current UI.
